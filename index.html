Contexto e Disciplina

Uma IA que colabora na engenharia de um projeto precisa de duas coisas que o código-fonte, sozinho, nunca entrega: contexto — saber o que o projeto é e por que existe — e disciplina — saber como se trabalha nele. A tese aqui é simples: o CLAUDE.md resolve o contexto; um ecossistema de Skills resolve a disciplina.

01 · Abstract

O problema não é escrever código. É saber onde e como.

Um modelo de linguagem é competente para gerar código, mas chega a qualquer projeto cego: não sabe o que o produto é, quais fronteiras não pode cruzar, onde cada coisa mora, nem qual é o padrão de qualidade da casa. Deixado por conta própria, ele inventa arquitetura plausível — e plausível não é correto.

Resumo

Este estudo descreve o processo de engenharia assistida por IA usado no Glenth/Luna — um monólito Laravel + Inertia/React organizado por Domain Design. O processo separa dois problemas ortogonais. O primeiro é o contexto: resolvido por um CLAUDE.md que funciona como fonte-de-verdade e índice de orientação, apontando para as camadas de produto (.project/) e de engenharia (.specs/).

O segundo é a disciplina: resolvida por um ecossistema de Skills vendorizadas — uma skill orquestradora (glenth-spec) que conduz um trilho Spec-Driven de sete fases e delega as disciplinas técnicas (TDD, debug, verification, code review), o commit e a documentação de produto a skills dedicadas.

O que torna o processo confiável não é a automação, e sim os seus breaking points: gates de controle onde o humano decide, onde o fluxo ramifica e onde há regras invioláveis (as Iron Laws). O restante do documento percorre as quatro camadas, disseca esses gates e descreve o que o processo entrega — e quando vale a pena adotá-lo.

As quatro camadas se empilham: (1) o índice de orientação, (2) o ecossistema de skills, (3) o trilho Spec-Driven e (4) as disciplinas técnicas delegadas. Sobre elas correm os gates.

02 · Camada 1 — Contexto

CLAUDE.md: fonte-de-verdade e índice de orientação

O CLAUDE.md não tenta conter o conhecimento todo — ele orienta. Diz o que a aplicação é (duas camadas: Platform e Domains), enuncia as regras invioláveis de arquitetura e, principalmente, aponta para onde a profundidade mora. Ele próprio se descreve como overview & index; AGENTS.md é apenas um symlink para ele.

A peça central é um mapa de documentação: uma tabela que roteia qualquer pergunta para a fonte certa, para que o agente nunca responda de memória quando existe um documento canônico.

Duas camadas de documentação, nunca confundidas

A distinção mais importante do repositório é entre o quê/porquê (produto) e o como (engenharia). Cada uma tem sua casa, seu dono e sua linguagem.

.project/ Produto

O quê e o porquê. Fractal: global (ecossistema) e por domínio. Linguagem de negócio, não-técnica — legível por um stakeholder.

  • vision.md · product.md · roadmap.md
  • docs/<feature>.md — regras de negócio + invariantes
  • technical-debit.md — única exceção técnica
  • changelog/AAAA/MM.md — só no global

Dona: skill glenth-docs.

.specs/ Engenharia

O como. Documenta o codebase e guarda as specs de cada feature. É a fonte-de-verdade técnica.

  • codebase/ — architecture, tech-stack, testing, auth-flow…
  • features/AAAA-MM-DD-<domínio>-<feature>/ — spec, design, tasks, context
  • project/STATE.md — memória persistente do processo

Dona: skill glenth-spec.

Identity questions go to the source, not to memory. CLAUDE.md — perguntas sobre o que o Glenth é (visão, produto, regras, roadmap) se respondem lendo o .project/ relevante, nunca de memória; nunca inventar uma visão quando o arquivo falta — dizer que é indocumentado.

Regra de idioma — deliberada

Toda documentação em .project/ e .specs/ é escrita em inglês (en_US) — visão, produto, roadmap, débito técnico, regras de negócio, changelog e specs de feature. A conversa com o dev e as mensagens de commit permanecem em PT-BR. A separação mantém os artefatos duráveis num idioma único e neutro, sem atritar com o diálogo do dia a dia.

03 · Camada 2 — Disciplina

Ecossistema de Skills: orquestração e delegação

Se o CLAUDE.md dá o contexto, as Skills dão o método. Elas são instruções empacotadas que o agente ativa conforme a tarefa. No Luna, a família glenth-* é vendorizada localmente em .claude/skills/ — sem dependência de plugin externo, nada a instalar.

Uma regra abre tudo: glenth-code-guidelines está SEMPRE ativa para qualquer trabalho de código — é o padrão de arquitetura e o rulebook (classes final, strict_types, thin controllers, Actions readonly, isolamento de domínios). Além dela, companheiras entram por tipo de tarefa.

O modelo de delegação

O ponto arquitetural das skills de processo: glenth-spec orquestra o trilho (o quê fazer e em que ordem) e delega o resto. As disciplinas técnicas ficam em skills dedicadas; o commit vai para glenth-commit; a documentação de produto vai para glenth-docs. O trilho permanece reutilizável, independente de quais disciplinas se aplicam.

04 · Camada 3 — Trilho

O fluxo Spec-Driven: sete fases, verificação por task

glenth-spec é um orquestrador leve: gates obrigatórios, zero cerimônia. O trilho tem sete fases — Specify → Design → Tasks → Execute → /simplify → Docs → Commit — e um princípio que atravessa todos os níveis: o floor. Antes de implementar qualquer coisa, registra-se Tipo + Motivo + Objetivo + Critérios de Aceite. Sem exceção, nem no modo Quick.

A profundidade se auto-dimensiona pela complexidade: um ajuste de três arquivos não paga o custo de design e tasks formais; uma feature multi-componente sim.

Auto-sizing por complexidade
EscopoCritérioTrilho
Small≤3 arquivos, 1 fraseQuick mode: implementar + verificar → /simplify → aceite → commit
MediumFeature clara, <10 tasksSpec breve → implementar + verificar → /simplify → aceite → docs → commit
LargeMulti-componenteSpec + IDs → design → tasks + deps → implementar/verificar por task → …
ComplexAmbiguidade, domínio novoSpec + discuss → research + arquitetura → tasks em paralelo → UAT → …
Plate 02Pipeline Spec-Driven com gates de controle
pedido→ contexto→ 01 Specify→ 02 Design→ 03 Tasks→ 04 Execute→ 05 /simplify→ suite→ Aceite (cego)→ 06 Docs→ 07 Commit→ commit atômico
Aceite REPROVADO → volta ao Execute (máx. 3 ciclos)
Commit = não → fica no working tree
Os blocos em âmbar são pontos de decisão/gate. O gate de aceite roda depois do /simplify e antes do commit; reprovação devolve o fluxo ao Execute.

As sete fases em detalhe

  1. S

    Specify obrigatório · floor

    Discovery (2–3 abordagens, perguntas uma de cada vez) e o registro do floor: Tipo, Motivo, Objetivo, Critérios de Aceite. Saída em spec.md com IDs rastreáveis [FEAT]-XX.

  2. D

    Design opcional

    Só quando há decisões arquiteturais. Apresentado incremental, seção por seção, com aprovação do dev. Saída em design.md.

  3. T

    Tasks opcional

    Quebra em tasks atômicas (What / Where / Depends / Done-when / Verify) sob a premissa "zero context". Pulado quando há ≤3 passos óbvios; se inline revela >5 passos, para e formaliza.

  4. E

    Execute obrigatório

    Implementação com TDD nas tasks com lógica, debug sistemático ao encontrar bug, e review de guidelines incremental por commit/task. Baseline de testes antes de começar.

    verificação executável por task — "Done when"
  5. /s

    /simplify obrigatório antes de commit

    Roda sobre o diff acumulado de todas as tasks, refatorando por simplicidade, escopo e intenção. A suite completa roda depois do /simplify — captura qualquer regressão introduzida pela refatoração.

  6. Dc

    Docs obrigatório em Medium+

    Triga glenth-docs: atualiza produto/roadmap/regras de negócio/changelog e destila o Objetivo + critérios de aceite nas invariantes do docs/<feature>.md. No Quick, é checklist inline.

  7. C

    Commit pergunta ao dev

    Nunca automático. Commits atômicos por domínio no formato <type>(<domain>|<scope>), cada um deixando o projeto rodável. Fechamento estruturado da branch em 4 opções.

    rastreabilidade Spec → Testes → Commit verificada

05 · Camada 4 — Disciplinas delegadas

Disciplinas técnicas: o rigor que o trilho invoca

O trilho define o quê e em que ordem. O como de cada disciplina técnica vive em skill própria, invocada pelo orquestrador na fase certa. Três delas carregam Iron Laws — regras invioláveis, escritas em caixa-alta, que não admitem atalho.

TDD
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
Ciclo RED → Verify RED → GREEN → Verify GREEN → REFACTOR. O teste que falha vem primeiro; ver o RED prova que o teste realmente exercita o comportamento.
Debug
NO FIXES WITHOUT ROOT CAUSE FIRST
4 fases (Root Cause → Pattern → Hypothesis → Fix) + defense-in-depth. Investiga-se a causa, não o sintoma; teste failing reproduz o bug antes do fix. Após 3 fixes falhos, escala ao dev.
Verify
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
Se você não rodou o comando nesta mensagem, não pode afirmar que passa. "Requirements met" ≠ "tests passing" — testar prova o que o teste afirma, não que a spec inteira foi atendida.

Code review — cinco eixos

A quarta disciplina não tem Iron Law, mas tem estrutura: todo review passa por Correctness · Performance · Maintainability · Security · Testing. E tem um protocolo para quando você é o autor do código revisado — ler completo, entender o argumento, verificar contra o código, avaliar, e só então responder concordando ou fazendo pushback com razão. Implementar feedback errado é pior que um bom pushback: zero performative agreement.

Correctness

O código faz o que a spec/comentário promete?

Performance

Hot paths, alocações, queries, N+1.

Maintainability

Naming, complexidade, duplicação, layering.

Security

Validação de input, secrets, authZ, injection — OWASP.

Testing

Cobertura, naming, anti-patterns de teste.

06 · O coração pedagógico

Breaking points: onde o humano decide e onde nada se cede

Automação sem gates é ilusão de produtividade. O que torna este processo confiável são os seus breaking points — pontos onde o fluxo para e ramifica. Eles se dividem em três naturezas: gates (o fluxo aguarda uma decisão ou uma validação), Iron Laws (regras invioláveis dentro de uma disciplina) e invariantes (verdades estruturais que nenhum passo pode violar).

Gate — o fluxo aguarda decisão/validação Iron Law — inviolável dentro da disciplina Invariante — verdade estrutural/de continuidade

07 · Resultado

O que o processo entrega — e quando vale a pena

A soma de contexto + disciplina + gates produz quatro resultados que se sustentam mutuamente. Não é velocidade bruta; é trabalho que não precisa ser refeito.

Documentação coerente

Produto (.project/) e engenharia (.specs/) permanecem sincronizados com o código, porque atualizá-los é um gate de encerramento, não um favor.

Commits rastreáveis

Atômicos, contextualizados por domínio, ligados aos IDs da spec — e cada um deixa o projeto rodável. O histórico conta a história certa.

Qualidade por evidência

Nada é "feito" sem output fresh; o aceite é julgado às cegas contra os critérios. Confiança sai da narrativa e vai para a prova.

Conhecimento acumulado

Decisões, overrides e lições ficam em STATE.md. Sessões novas retomam de onde a anterior parou, sem reconstruir contexto.

A cadeia de rastreabilidade

O fio que amarra tudo é Spec → Testes → Commit: cada critério QUANDO/ENTÃO da spec.md gera um teste nomeado com o ID do requisito; antes do commit, verifica-se que todo [FEAT]-XX tem código e teste passando. Um critério sem evidência é um FAIL — não um "provavelmente ok".

Quando isto vale a pena

O processo tem custo de cerimônia. Ele se paga quando o projeto é longevo, multi-domínio e trabalhado em muitas sessões — exatamente o caso do Luna. Em trabalho descartável, ele pesa mais do que ajuda. A régua honesta para o leitor decidir adotar:

Vale a pena quando

  • O código vai viver e evoluir por meses ou anos.
  • Há fronteiras arquiteturais que não podem erodir.
  • Mais de uma pessoa (ou sessão) toca a base.
  • Regressões silenciosas custam caro.
  • A IA é colaboradora recorrente, não pontual.

Pesa demais quando

  • É um spike, protótipo ou script de uso único.
  • O código será jogado fora em dias.
  • Não há domínios nem invariantes a proteger.
  • Exploração livre importa mais que rastreabilidade.

Adotar não é tudo-ou-nada. O auto-sizing existe justamente para isto: comece pelos gates que mais doem no seu contexto — autorização de commit, evidência antes da afirmação, um índice de orientação — e deixe o trilho crescer conforme o projeto pede.
